# paperplot 不足总结：实战中的痛点与边界

在高强度、多维度的 JCIS 数据绘图任务中，我也发现了一些 `paperplot` 目前存在的局限性和待改进之处。

## 1. 静态颜色循环 (Color Cycle) 的瓶颈
*   **重复率高**：`yelan` 等主题内部预设的颜色数量较少（如 3 种）。在绘制包含 5 个或更多实验条件的分组柱状图（如 `Add2`）时，颜色会发生自动循环，导致不同标签的颜色完全相同，丧失了基本的视觉区分功能。
*   **手动干预繁琐**：目前必须通过底层 `ColorManager` 或 `set_prop_cycle` 才能覆盖这一行为，缺乏一个高层级的“调色板自动扩展”接口。

## 2. 布局间距控制 (Padding API) 的局限性
*   **维度缺失**：目前的 `set_padding` 方法仅提供了四个边的外边距（left, right, top, bottom），却缺失了最关键的内部间距（`hspace` 和 `wspace`）的参数化控制。
*   **标题与图例的物理冲突**：当启用全局标题（`suptitle`）和全局图例（`legend`）时，系统没有自动冲突检测机制。在 4x1 或 5x1 这种细长布局中，标题几乎总是会盖住图例，必须依靠开发者不断手动调整 `y` 和 `top` 的数值。

## 3. 坐标轴属性与数据类型的隐含推断问题
*   **柱状图消失迷云**：如果传入的 X 轴数据（如时间 0, 30, 60...）是数值型，`paperplot` 内部（通过 matplotlib）有时会将其作为连续数值轴处理，导致离散柱子的宽度被压缩至消失。
*   **类型转换负担**：开发者必须养成手动将数据 `.astype(str)` 的习惯，库层级缺乏一个强制分类坐标轴（Categorical Axis）的开关。

## 4. 底层与高层 API 的耦合报警
*   **布局引擎冲突**：在使用 `Plotter` 默认的布局时，手动调用 `subplots_adjust` 经常会触发 `UserWarning` 或导致布局崩坏。这意味着库的“全自动布局”与开发者的“精细手工微调”之间尚未达成完美的协议。

## 5. 错误提示的友好度
*   **标签未定义时的困惑**：当在一个复杂的 `layout` 中忘记为某个 `tag` 绘图却尝试对其设置属性时，报错信息有时较为晦涩，初学者难以快速定位是布局定义问题还是代码逻辑问题。

---
**总结**：`paperplot` 目前在“全自动”和“全控制”之间还存在一些灰色地带，尤其是在处理极大规模（多子图、多系列）复合图表时，仍需开发者具备较强的底层调试能力。
